# Weinstein & Swing Pro Dashboard v67.4.12 — User & Trading Guide

> **Module Role:** The **mission control panel** of the entire Weinstein Commander ecosystem. The Dashboard is the single indicator you keep open at all times — it synthesises data from every other module (stage, RS, momentum, alpha score, catalysts, price-action structure, portfolio P&L) into one unified on-chart table. It does not signal entries on its own; rather, it **validates and contextualises** the signals generated by the Strategy and Screeners.

### What this means for trading

1. **Trust the Tier +3 states blindly** — STAGE_2_LAUNCH, GAP_UP_BO, VCP_BO, SPRING all show statistically airtight alpha across thousands of trades. These are the highest-edge setups in the field.
2. **BULL_ENGULF is now selective** — only fires on bars where the engulf occurs at RSI(14) < 40 with relvol > 2.0 (institutional volume + oversold extreme). When it fires, take it seriously.
3. **Watch states are advisory only** — DISTRIBUTION_DAY / FAILED_BREAKOUT / SHOOTING_STAR_RESIST / BEAR_ENGULF still display but no longer penalize the Action Signal. Use them as discretionary read-outs, not as automatic vetoes.
4. **Co-firing bullish states win** — if a bar fires both DISTRIBUTION_DAY and POCKET_PIVOT (or any bullish state), the cascade now resolves to the bullish state. Previously DISTRIBUTION_DAY would override — that behaviour was actively wrong per the data.
5. **Late-stage bases get demoted automatically** — B3 breakouts demoted −1 tier, B4+ demoted −2 tiers. Stage-2 base #1 is prime; #3+ is late cycle.
6. **Volume tag at a glance** — "(Vol++)" = institutional ≥2.0×, "(Vol+)" = confirmed ≥1.5×, "(Low Vol)" = warning <0.7×.

### Timeframe scope (added 2026-05-24)

**The PA cascade was validated on the DAILY timeframe only.** The dashboard renders on any chart timeframe, but:
- **Daily** — fully validated; per-state alphas above apply
- **Intraday (125-min, 75-min, etc.)** — daily/weekly structural references (Stage, MAs, RS) still correctly fetch from higher TF, but candle-pattern detectors (`pa_hammer`, `pa_nr7`, `pa_vcp_bo`, etc.) operate on chart-TF bars. The alpha numbers above DO NOT TRANSFER to intraday; an intraday `pa_vcp_bo` is a fundamentally different event than a daily one
- **Weekly** — not designed for; mixed-TF references create odd mismatches

**Recommended usage:** keep the PA field on daily for setup discovery and positional decisions. For intraday entry timing, switch to chart timeframe and read raw price action manually rather than trusting the field's tier outputs.

### Workflow recipe

To take a trade decision:

1. **Load the Dashboard** on the candidate chart (Decision Mode). Read top-down:
   - Is `RECOMMENDATION` actionable (BUY / STRONG BUY)?
   - Is `ACTION SIGNAL ≥ 7/10`?
   - Does `CATALYST + GATES` show a valid catalyst with green gates?
   - Is `ASSET QUALITY ≥ 60`?

2. **If a catalyst is firing**, open Bull Screener (for POS / SWG) or Recovery Screener (for REV) to read:
   - The detailed gate status (G1…G9 / G1…G10)
   - The pivot-based trade levels (SL / T1 / T2)
   - The "How to Trade" guidance

3. **For multi-engine validation**, glance at Unified Ecosystem to confirm both Bull and Recovery engines agree (or at least don't conflict).

4. **For capitulation plays**, Recovery Screener is the primary surface — Dashboard's RS row provides the structural context.

You should never need to read the same metric in two places. If you do, one of them is out of sync — file a sync report.

## What's New in v67.2.3 — Daily Trend Cross-Tool Alignment

The Dashboard's `Daily Trend` row was disagreeing with the on-chart **Zigzag [Strict]** panel for the same bar (Dashboard said DOWN; Zigzag said SIDEWAYS). Both tools share the same v6.2 strict-trend engine — the disagreement was purely a settings-default mismatch:

- v67.2.1 bumped daily-pivot defaults from 2/2 → 3/3 (intended to reduce strict-trend noise).
- The on-chart Zigzag indicator ships with 2/2 defaults.
- Different pivot windows detect different swings, so the two engines produced different trend states for the same data.

**Fix:** Daily pivot defaults rolled back to **2/2** so Dashboard and Zigzag agree out-of-the-box. Users who preferred the smoother 3/3 view can still set it via the input. The tooltip now flags the "must match on-chart Zigzag" dependency loudly. See new §14 for the full cross-tool alignment checklist.

## What's New in v67.2.2 — Stage 2 Age Disambiguation

Three different tools in the ecosystem were reporting three different "Stage 2" durations for the same stock — they were all correct, but measured different things sharing the same label. v67.2.2 surfaces both anchors side-by-side and adds a categorical freshness tag.

- **Market Structure** row no longer carries a week count — it's now purely the structural verdict.
- **New row** below it — **`Stage 2 Age`** — shows `<leg>w leg / <macro>w macro · <Fresh|Maturing|Extended>` whenever stage is Stage 2.
  - **leg** = sub-stage age, resets on every UP↔PULLBACK transition (matches the Dashboard's historical week count)
  - **macro** = full Stage-2 lifespan, only resets when the stage family exits Stage 2 entirely (matches the Commander Bull Screener's `(47w)` style reading)
  - **Fresh** when leg ≤ `stage2_fresh_weeks` (default 13w) — matches the Unified Ecosystem's freshness flag
  - **Maturing** when macro ≤ 26w
  - **Extended** when macro > 26w
- New input under **Advanced Thresholds → Stage 2 'Fresh' Threshold (weeks)** (default 13) to tune the Fresh/Maturing cutoff.
- Why now: the long-standing disconnect — Dashboard showing 12w, Screener showing 47w, Ecosystem saying "Fresh" — was a semantic overload, not a bug. See §13 for the full explanation.

## 2. Input Groups — Field-by-Field

### Groups 1–6: Portfolio Slots (1–30)

> **CRITICAL — Portfolio slots are for OPEN positions only.**
> Dashboard portfolio slots 1–30 are reserved exclusively for **stocks you currently hold** (i.e. positions actually in your Dhan account). They are NOT meant for watchlist tracking or for stocks you're considering. Watchlist stocks belong in TradingView's native watchlist (or in `Generated_Watchlists/` files produced by the Commander Web's "Run Full Auto-Pilot" workflow — see `07_Commander_Web_v4_Guide.md` §2 Tab 4 + Bull/Recovery watchlist files). The portfolio slot machinery is purpose-built for open-position bookkeeping: P&L rows, R-multiple, days-held, time-stop warnings, sector rotation alerts, ATR-trailed stop tracking — all of which only make sense once a position is live with a defined entry, SL, and entry date.
>
> **The `master_portfolio_sync.py` script enforces this.** It pulls only the positions currently open in your Dhan account, never watchlist symbols. If you manually populate a slot with a watchlist ticker you don't own, the P&L / R-multiple / days-held metrics will lie to you.

The dashboard supports up to **30 live portfolio positions** organised in groups of 5 slots each. Each slot has:

| Sub-Input | Explanation |
|---|---|
| **Slot N (ticker)** | The NSE/BSE ticker, e.g. `NSE:RELIANCE`. Used to match the current chart ticker to a portfolio entry. |
| **Entry (price)** | Your average entry price in INR. Used to compute unrealised P&L and R-multiple. |
| **SL (stop-loss price)** | Your active stop-loss price. Used to compute distance-to-stop and trigger warnings. |
| **Sector (symbol)** | The sector index for this holding (e.g. `NSE:CNXENERGY`). Used in sector RS comparison. The Python automation (`master_portfolio_sync.py`) fills this from `sector_manager.py`. |
| **Date (input.time)** | Entry date. Enables the Days Held counter and Time Stop warning. Set using the `input.time` picker or injected by `master_portfolio_sync.py`. |

**How the match works:** On each chart load, the script calls `f_match(ticker)` which strips the exchange prefix and compares against `syminfo.ticker`. When a match is found, that slot's entry, SL, date, and sector become the "active position" data for the current chart.

**Quick Check override:** Two additional inputs (`quick_ent` and `quick_sl`) allow temporary override for analysing a potential trade without changing any portfolio slot.

## 4. The Recommendation Logic

The **RECOMMENDATION** cell is the headline verdict — derived from Stage + Daily Trend + RS Level + Market Health.

```
STRONG BUY (Momentum)   : Stage 2 (UP) + Daily Uptrend + RS Positive + Market not Bearish
BUY (Check RS)          : Stage 2 (UP) + Daily Uptrend          + Market not Bearish
HOLD (Consolidation)    : Stage 2 (UP) + Daily Sideways
WATCHLIST (Base)        : Stage 1 (BASE)
CAUTION (Divergence)    : Bearish RSI divergence active
NO TRADE (Downtrend)    : Daily trend = DOWN
AVOID (Stage 4)         : Stage 4 (DOWN)
WAIT (Mkt Bearish)      : Any signal degraded by Nifty 500 in BEARISH regime
EXIT NOW (40WMA Break)  : Hard panic exit — close < 40-week MA on the weekly
```

> **`BUY*` removed in v67.0.** The RFF Lite fundamental gate and its downgrade behaviour migrated to the Weinstein Unified Ecosystem indicator. The Dashboard's Recommendation cell now reflects technical state only; pair it with the Unified Ecosystem panel for the fundamental overlay.

### How to combine RECOMMENDATION with ACTION SIGNAL

`RECOMMENDATION` tells you **what stage/trend regime says** (the "should I be looking at this stock?"). `ACTION SIGNAL` tells you **how confluent today's bar is** (the "is now the moment?"). Read them together:

| Recommendation | Action Signal | Interpretation |
|---|---|---|
| STRONG BUY | ≥ 7/10 | Full confluence — act on the next valid pivot |
| STRONG BUY | 4–6/10 | Right regime, wrong day — wait for tighter bar |
| BUY | ≥ 7/10 | Acceptable entry; size down vs STRONG BUY |
| HOLD / WATCHLIST | ≥ 7/10 | High-quality bar inside a base — pocket-pivot or coil; add to watchlist with alerts |
| AVOID / NO TRADE | any | Skip regardless of bar quality |
| any | ≤ 2/10 | Wait — even a clean stock has zero edge today |

## 8. Daily Usage Workflow

### Pre-Market (~15 mins)
1. Load the Dashboard on the current chart
2. Check **Mkt Health (N500)** row → if `BEARISH`, scale position sizes down by 25% and tighten T1/T2 (dashboard auto-compresses these via `dyn_T1_R/T2_R/T3_R` profile)
3. Check **Sector Stage (W)** → if sector in Stage 3/4, skip the stock regardless of individual signals
4. Read **Market Structure** → only act on Stage 2 (UP / Pullback) or Stage 1 (Constructive) — never Stage 3/4
5. Read **RECOMMENDATION** → STRONG BUY or BUY = proceed to the Action Signal gate; AVOID/EXIT NOW = stop reading

### During-Market (signal management)
1. Load any candidate from the Screener on the chart
2. Verify **ASSET QUALITY ≥ 60** (B grade or better)
3. Read **ACTION SIGNAL** — only fire on **≥ 7/10 (Strong)** for breakouts, **≥ 5/10 (Good)** for pullbacks
4. Cross-check **Price Action Structure** — A+/A tier setups (★ VCP BREAKOUT, ★ POCKET PIVOT, ★ SPRING, BREAKOUT Confirmed) carry the highest reward-to-effort
5. Check **EMA20 Proximity** → if "OverExtended," wait for pullback (or scale entry size down)
6. Check **Room for Trade** → if "⚠️ NO ROOM," abandon swing setups; allow positional with caveats
7. For an open position, check **Time Stop (P/S)** — if stagnant, follow the time-stop rule

### Post-Market (~10 mins)
1. Update portfolio slot data if any new trades were taken
2. Run `master_portfolio_sync.py` to auto-sync Dhan positions into the dashboard Pine inputs
3. Review **PORTFOLIO HEALTH** row for W/L/Stop tally and review trades flagged TRIM or STOP HIT

## 9. Common Mistakes

1. **Ignoring Action Signal when Recommendation is green** — STRONG BUY at 3/10 is a "right stock, wrong day" situation. The signal is built to filter exactly this.
2. **Leaving Quick Check values populated** — `quick_ent` / `quick_sl` override portfolio data for the matched ticker. Always reset to 0 after analysis.
3. **Using wrong sector in portfolio slot** — If you set the wrong sector ticker in a slot, the RS vs. Sector row will be meaningless. Let the Python automation or the auto-mapper set this.
4. **Ignoring Time Stop** — The time stop is a discipline tool. A trade open 10+ days below 0.5R is a dead-money situation; the warning exists to force a decision.
5. **Not syncing Entry Date** — Without a correct date in the slot's `input.time` field, the Days Held counter and Time Stop warning cannot function. Always set the date when entering a position.
6. **Misreading paTier-negative setups** — `⚠️ FAILED BREAKOUT`, `DISTRIBUTION DAY`, and `BEARISH ENGULFING` actively *subtract* from the Action Signal. A stock with full daily confluence (6/6 raw) but a failed-BO bar lands at 4/10 — the dashboard is warning you, not just being cautious.

## 11. Action Signal — How the Score Is Computed

The **ACTION SIGNAL** cell displays `X/10 (Label)` where `X` is the integer composite score:

```
action_score = clamp( raw_conf + paTier , 0 , 10 )
```

**raw_conf (0–6)** — six boolean confluence factors, +1 each when true:

| Factor | Condition |
|---|---|
| `c_trend` | close > EMA20 |
| `c_mom` | daily RSI > 50 |
| `c_vol` | Smart vol: up-bar with vol > avg, OR down-bar with vol < avg |
| `c_vp` | close > high of the max-volume bar in last `vpLookback` (Max-Vol Bar / POC proxy) |
| `c_adx` | ADX > 20 with +DI > −DI |
| `c_liq_sweep` | SMC liquidity sweep + reclaim of SMA50 with vol > 1.5× (gated by `liq_sweep_lockout`) |

**paTier (−2 to +4)** — single integer from the Price Action cascade (§10). Carries far more weight than any individual boolean: a `★ VCP BREAKOUT` adds +3 by itself, while a `⚠️ FAILED BREAKOUT` deducts 2.

### Label thresholds

| Score | Label | Color |
|---|---|---|
| 9–10 | **Excellent** | Bull green |
| 7–8 | **Strong** | Bull green |

### Worked examples

**Stock A** — Stage 2, daily uptrend, RS positive, fresh VCP breakout on volume:

```

```

The Recommendation cell may still say **BUY** (Stage 2 + uptrend + RS positive) — but the Action Signal correctly drops to "Moderate" and the cell colour goes orange. **That's the coupling working as designed.**


A high-quality bar on a broken stock still scores Weak — the system correctly refuses to act on isolated bullish bars in a hostile context.

## 13. The Three "Stage 2" Numbers — Resolved in v67.2.2

A long-standing source of confusion: for the same Stage-2 stock, three tools in the ecosystem reported three different durations:

| Tool | Example | What it actually measured |
|---|---|---|
| Dashboard (≤ v67.2.1) | `STAGE 2 (Full Confluence) (12w)` | Sub-stage **leg age** — resets every time the stock dips below the 30 WMA and recovers (UP ↔ PULLBACK transitions) |
| Commander Bull Screener v2.9 | `Stage 2 (UP) (47w)` | **Macro Stage-2 lifespan** — time since first entry into the Stage 2 family; pullbacks don't reset it |
| Unified Ecosystem v2.5 | `Stage 2 Freshness: Fresh` | **Categorical tag** keyed off the leg age (typically ≤ 13w) |

All three were computed correctly. The problem was semantic — the same label "Stage 2" was carrying three different time-anchors across modules.

### Why all three are useful

| Metric | Decision it informs |
|---|---|
| **Leg age** (12w) | Entry timing — fresh advance legs after a healthy pullback are the highest-quality entries |
| **Macro age** (47w) | Position sizing & risk — older Stage-2 trends carry exhaustion risk, even if today's bar looks clean |
| **Fresh / Maturing / Extended tag** | Watchlist filter — surface stocks resuming Stage 2, not late-stage runners |

### How v67.2.2 fixes it

`f_getStageAndTime` now tracks a second anchor — `macroStart` — that only resets when the stage family fully exits Stage 2 (to Stage 1 / 3 / 4). The bundle pipes both anchors through, and the dashboard renders them in a single dedicated row:

```
Market Structure   🚀 STAGE 2 (Full Confluence)
Stage 2 Age        12w leg / 47w macro · Fresh
```

So on the same chart that previously showed `(12w)`, you now see — at a glance — that the current advance leg is fresh (12w), but the trend itself has run for 47 weeks (cross-checks against the Screener). The colour of the `Stage 2 Age` cell tracks the tag: green = Fresh, orange = Maturing/Extended.

### Reading combinations

| Leg | Macro | Tag | Practical reading |
|---|---|---|---|
| ≤ 13w | ≤ 13w | Fresh | Brand-new Stage 2 launch — best risk/reward |
| ≤ 13w | 14–40w | Fresh | Resumption after pullback — same risk/reward as fresh launch if base was clean |
| 14+w | 14–26w | Maturing | Trend has legs; entries OK but size down vs. the above |
| 14+w | 27+w | Extended | Late-stage trend — only act on highest-tier PA setups (paTier ≥ +3), expect chop |

### Side-effects audited & preserved

- The **Auto-Anchored VWAP** still uses the *sub-stage* anchor (`stageStart`), not the new macro anchor. AVWAP is meant to anchor the most recent advance leg — that behaviour is intentionally unchanged.
- The **Recommendation cell** is regime-only and doesn't read the week count; no behavioural change there.
- **Action Signal** doesn't read week counts either; no behavioural change.

Net effect: zero changes to entry/exit signalling — the only thing that changed is the dashboard now answers all three "how long has this been in Stage 2?" questions at the same time instead of forcing you to remember which tool means what.

## 14. Architectural Principle — Zigzag Is the Source of Truth (v67.2.4)

### The principle

**The on-chart `Zigzag [Strict v6.2]` indicator is the canonical source of truth for HH / HL / LH / LL classification and the derived daily/weekly trend state.** The Dashboard's `f_getStrictTrend`, `f_classifyHigh`, and `f_classifyLow` functions are a **mirror** of Zigzag's internals — they must remain bit-identical to the canonical engine. No independent evolution.

This is captured in the architectural-contract header at the top of the Pine file:

```
============================================================================
 STRICT-TREND ENGINE — ARCHITECTURAL CONTRACT
============================================================================
 The canonical source of truth for HH / HL / LH / LL classification and the
 derived trend state is the Zigzag [Strict] indicator (v6.2) — NOT this
 Dashboard. The functions f_classifyHigh / f_classifyLow / f_getStrictTrend
 below are a mirror of Zigzag's internals and must remain bit-identical to
 it. If Zigzag's strict engine changes, this file MUST be updated to match —
 no independent evolution.
============================================================================
```

### Why this matters

Before v67.2.4 the Dashboard had its own strict-trend code, manually kept "in sync" with Zigzag via changelog notes. That works until it doesn't — settings drift (the 3/3 vs 2/2 incident in v67.2.1→.3), or one tool gets a bug fix the other doesn't (v6.0 vs v6.2 across files). Making Zigzag canonical means:

- **Daily Trend / Master Trend rows are reliable** — they represent the same engine output you see in the Zigzag chart panel.
- **Maintenance has a single direction of flow** — update Zigzag, then propagate to the Dashboard mirror. Never the reverse.
- **Drift is detectable at a glance** — the new `Trend Engine` row turns orange the moment a knob is overridden.

### The three knobs that must match

| Knob | Where it lives | Canonical default | What it controls |
|---|---|---|---|
| **Engine version** | Source code of each indicator | v6.2 | Pivot classification + extension path + strict gate logic. v6.0 → v6.2 fixed four real bugs: raised `eq_threshold` 0.1 % → 0.5 %; extension path classifies against `prevLockedHigh/Low`; EH/EL no longer confirm trend; opposite-side reads confirmed class. |
| **Pivot Left/Right** | User input | **2/2** (daily) — 5/5 hard-coded weekly | Window for detecting swing highs/lows. Different windows = different pivots = different trend states. |
| **EH/EL threshold** | `eq_threshold_dash` (Dashboard) / `eq_pct` (Zigzag) | **0.005 (0.5 %)** | Tolerance for classifying a near-equal high/low as "EH" / "EL" instead of "HH" / "LL". |

### Reading the Trend Engine row

The Dashboard's `Trend Engine` row renders one of two states:

| State | Display | Meaning | What to do |
|---|---|---|---|
| ✅ Canonical | `v6.2 · L2/R2 · 0.50%` (GREEN cell) | All three knobs at Zigzag defaults | Nothing — Daily Trend cell can be trusted to match the on-chart Zigzag. |
| ⚠ Drifted | `v6.2 · L3/R3 · 0.50% ⚠ Sync Zigzag` (ORANGE cell) | At least one knob overridden | Apply the **same** override to the Zigzag indicator's settings, then both tools agree again. |

The cell does NOT turn red — drift is a warning, not a failure. You're allowed to run non-canonical settings; you just have to apply them consistently across tools.

### 60-second alignment check (when in doubt)

1. **Open the on-chart Zigzag panel.** Read its `CONFIG` row — note pivot `L=n R=n` and threshold.
2. **Glance at the Dashboard `Trend Engine` row.** If GREEN → already aligned with canonical Zigzag defaults. If ORANGE → compare its values against Zigzag's CONFIG row.
3. **Compare the displayed states** — Dashboard `Daily Trend (D)` ↔ Zigzag panel `TREND` should match.

If both knob sets match and the displayed trends still disagree, then the algorithm itself has drifted (one file is on a different engine version). Diff `f_getStrictTrend`, `f_classifyHigh`, `f_classifyLow` between the two `.pine` files — they must be character-identical.

### Maintenance rules (for the maintainer)

1. **Never edit `f_getStrictTrend` / `f_classifyHigh` / `f_classifyLow` in the Dashboard first.** Edit the Zigzag indicator first, validate it on charts, then port the change verbatim into the Dashboard.
2. **The Dashboard's `Trend Engine` defaults (2/2, 0.5 %) must match Zigzag's defaults.** If Zigzag ever changes its defaults, the Dashboard's `pivotLeft`, `pivotRight`, and `eq_threshold_dash` defaults must be updated in the same commit.
3. **Bump the engine version literal in two places** when porting: the architectural-contract header at the top of the Pine file, and the `trend_engine_str` string that drives the dashboard row.
4. **The changelog entry for any strict-trend port should explicitly state the Zigzag commit / version it was synced from.**
